# 限流配置最佳实践

本文档介绍如何在 fastapi-tp6 项目中使用 `ApiRateLimit` 注解，合理配置限流策略以保护系统稳定性。

## 1. 限流适用场景

### 1.1 需要限流的场景

- **认证接口**：登录、注册、验证码（防止暴力破解）
- **敏感操作**：密码修改、删除操作（防止恶意操作）
- **资源消耗接口**：上传、导出、复杂查询（保护服务器资源）
- **公开接口**：无需认证的 API（防止滥用）

### 1.2 限流维度选择

| 维度 | 适用场景 |
|------|----------|
| `ip` | 匿名接口、公开 API |
| `user` | 已登录用户的操作接口 |
| `user_or_ip` | 混合场景（登录按用户，未登录按 IP） |

## 2. 预设配置使用

### 2.1 匿名认证接口

```python
from app.annotations import ApiRateLimit, ApiRateLimitPreset

# 登录接口（10次/分钟）
@ApiRateLimit(namespace='login', preset=ApiRateLimitPreset.ANON_AUTH_LOGIN)
async def login(request: Request, username: str, password: str):
    return await AuthService.login(username, password)

# 注册接口（5次/2分钟）
@ApiRateLimit(namespace='register', preset=ApiRateLimitPreset.ANON_AUTH_REGISTER)
async def register(request: Request, user_data: dict):
    return await AuthService.register(user_data)

# 验证码接口（30次/分钟）
@ApiRateLimit(namespace='captcha', preset=ApiRateLimitPreset.ANON_AUTH_CAPTCHA)
async def get_captcha(request: Request):
    return await AuthService.generate_captcha()
```

### 2.2 用户操作接口

```python
# 普通修改操作（20次/分钟）
@ApiRateLimit(namespace='update', preset=ApiRateLimitPreset.USER_COMMON_MUTATION)
async def update_profile(request: Request, data: dict):
    return await UserService.update_profile(data)

# 安全敏感操作（10次/分钟）
@ApiRateLimit(namespace='security', preset=ApiRateLimitPreset.USER_SECURITY_MUTATION)
async def change_password(request: Request, old_pwd: str, new_pwd: str):
    return await UserService.change_password(old_pwd, new_pwd)

# 删除操作（5次/分钟）
@ApiRateLimit(namespace='delete', preset=ApiRateLimitPreset.USER_DESTRUCTIVE_MUTATION)
async def delete_account(request: Request):
    return await UserService.delete_account()
```

### 2.3 通用接口

```python
# 上传接口（10次/分钟）
@ApiRateLimit(namespace='upload', preset=ApiRateLimitPreset.COMMON_UPLOAD)
```

## 3. 自定义限流配置

### 3.1 覆盖预设参数

```python
@ApiRateLimit(
    namespace='login',
    preset=ApiRateLimitPreset.ANON_AUTH_LOGIN,
    limit=5,            # 覆盖预设的 limit
    window_seconds=30   # 覆盖预设的窗口时长
)
async def strict_login(request: Request):
    # 更严格的登录限流
    return await AuthService.login()
```

### 3.2 完全自定义

```python
@ApiRateLimit(
    namespace='custom_api',
    limit=50,
    window_seconds=60,
    scope='ip',
    algorithm='fixed_window',
    fail_strategy='local_fallback',
    message='请求过于频繁，请稍后再试'
)
async def custom_api(request: Request):
    return {'data': 'ok'}
```

## 4. 故障策略选择

### 4.1 策略对比

| 策略 | Redis 不可用时行为 | 适用场景 |
|------|--------------------|----------|
| `open` | 放行请求 | 非关键接口、用户体验优先 |
| `closed` | 直接拦截 | 关键接口、安全优先 |
| `local_fallback` | 进程内存限流 | 单实例兜底 |

### 4.2 推荐配置

```python
# 登录接口：使用 local_fallback（即使 Redis 不可用也要限流）
@ApiRateLimit(namespace='login', preset=ApiRateLimitPreset.ANON_AUTH_LOGIN)
# 预设默认 fail_strategy='local_fallback'

# 查询接口：使用 open（用户体验优先）
@ApiRateLimit(namespace='query', limit=100, fail_strategy='open')

# 删除接口：使用 closed（安全优先）
@ApiRateLimit(namespace='delete', preset=ApiRateLimitPreset.USER_DESTRUCTIVE_MUTATION, fail_strategy='closed')
```

## 6. 限流算法

### 6.1 固定窗口算法

当前使用固定窗口算法：

- 将时间划分为固定窗口（如 60 秒）
- 每个窗口内独立计数
- 窗口结束时计数清零

**特点**：
- 实现简单，性能高
- 可能存在边界突发问题

### 6.2 未来支持

预留 `sliding_window` 滑动窗口算法参数，后续可扩展支持更精确限流。

## 8. 常见问题

**Q: 如何查看当前限流状态？**

查看响应头：
- `X-RateLimit-Limit`：窗口内最大次数
- `X-RateLimit-Remaining`：剩余次数
- `X-RateLimit-Reset`：重置时间（秒）

**Q: 用户维度限流时未登录会怎样？**

- `scope='user'`：未登录用户跳过限流
- `scope='user_or_ip'`：未登录用户按 IP 限流

**Q: 多实例部署时 local_fallback 有效吗？**

仅保证单实例内生效。多实例建议依赖 Redis。

**Q: 如何处理限流触发后的前端提示？**

前端捕获 HTTP 429，根据 `Retry-After` 头显示等待时间。
