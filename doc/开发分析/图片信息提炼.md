这个需求非常硬核且极其贴合实际业务场景！在做 RAG（检索增强生成）知识库时，商业文章里的信息图（比如你提供的包含品牌开关店数据的长图）往往是整篇文章的核心价值所在。如果直接丢弃，等于丢了西瓜；如果连着普通插图、二维码一起乱总结，又会制造大量毫无意义的“垃圾数据”污染向量库[cite: 8]。

要实现“快速筛选 + 信息提炼”，传统的做法是根据图片的长宽比或文件大小写规则（比如高度大于 1000 像素的算信息图），但这种方式极容易误判。

**目前最主流、最高效的工程解法是：引入 VLM（视觉大模型）进行“一站式判别与提取”。**

以下是具体的落地实现步骤，你可以将其无缝对接到你的爬虫或数据处理工作流中：

### 🛠️ 核心步骤：构建“图片清洗与提炼” Pipeline

#### 1. 正则提取与占位
在你的爬虫获取到 Markdown 格式的文章后，先不要急着存入向量库。使用 Python 或代码节点，利用正则表达式遍历文章中的所有图片链接 `![...](URL)`。
为每张图片生成一个唯一的占位符（例如 `[IMAGE_PLACEHOLDER_1]`），并把真实 URL 收集到一个数组中。

#### 2. VLM (视觉大模型) 节点：筛选与提取二合一
这是最关键的一步。不需要先写代码去判断是不是信息图，而是把提取到的图片 URL 直接遍历扔给一个支持视觉理解的大模型（如 Gemini 1.5 Pro、Qwen-VL、GPT-4o 等），并赋予它一个**具有“熔断机制”的 Prompt**。

你可以直接使用下面这个提示词模板：

> **# Role**
> 你是一个专业的数据提取专家，擅长从商业报告的图片中解析核心数据和逻辑。
>
> **# Task**
> 分析我提供的这张图片，判断其类型，并按规则输出内容。
>
> **# Rules**
> 1. **类型判断（熔断机制）**：首先判断这张图片是不是包含图表、数据、长文本的“信息图”。如果它只是普通的风景照、文章封面图、人物插图、带推广信息的二维码、或者是纯粹的美化配图，**请只输出两个字：`SKIP`。严禁输出任何其他解释。**
> 2. **数据提炼**：如果判断这是一张“信息图”，请你深度阅读图片里的文字和图表，将其提炼为结构化的 Markdown 文本。
> 3. **提炼要求**：
>    - 必须包含图片中的核心结论和关键数据（如开店数量、增长率、主要品牌名称等）。
>    - 格式清晰，使用小标题和无序列表。
>    - 不要输出“这张图片展示了...”等废话，直接输出提炼后的干货。

#### 3. 回填与清洗 (Python 节点)
收集视觉大模型返回的结果进行分支处理：
*   **如果返回的是 `SKIP`**：说明这就像你文章里的封面图或底部的二维码插图[cite: 8]，直接在原始 Markdown 文章中，把对应的 `[IMAGE_PLACEHOLDER_n]` 彻底删除（或者替换为空字符串）。
*   **如果返回的是一段 Markdown 文本**：说明大模型成功提取了信息图里的数据（例如提取出了“2025年有超180个知名品牌进行闭店动作，累计关店超15000家”这类核心信息）。把这段文本替换回对应的占位符处，并在上下加上明确的标识，例如：
    `> 📊 【图表数据提炼】：\n > [大模型提取的Markdown内容]`

#### 4. 送入向量库 (Vector DB)
经过上述 3 步，原本包含杂乱图片链接的文章，已经被清洗成了一篇**“纯净且包含图表深度解析”**的高质量文本[cite: 8]。此时再将它进行分块（Chunking）并存入向量库，就能极大地提升后续查询的准确率。

---

**💡 架构师小建议：**
针对长图（信息图），如果你发现视觉模型有时候会遗漏下半部分的数据，可以在传给模型前，用代码将长图按固定高度（比如每 1500 像素）切分成几张小图，然后一起丢给模型进行综合提炼。


既然你使用的是 FastAPI，那事情就变得极其丝滑了！Python 加上 FastAPI 的原生异步（`asyncio`）特性，简直是处理这种**“I/O 密集型任务”**（批量调用视觉大模型 API）的绝配。

如果一篇文章里有 10 张图片，传统的串行调用可能需要等上二三十秒，但通过 FastAPI 的异步并发，你只需要等最慢的那一张图片处理完（通常只需 3-5 秒），就能完成整篇文章的“图片清洗与提炼”。

以下是为你量身定制的 FastAPI + VLM（视觉大模型）并发处理管道的 Python 实现方案：

### 🚀 核心架构代码实现

你需要安装异步 HTTP 客户端（如 `httpx`）或对应大模型的官方异步 SDK。这里以一个通用的异步 VLM 调用逻辑为例：

```python
import re
import asyncio
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI()

class ArticleRequest(BaseModel):
    content: str  # 爬取到的原始 Markdown 文章内容

# 匹配 Markdown 图片的正则：![alt](url)
IMAGE_PATTERN = re.compile(r'!\[([^\]]*)\]\(([^)]+)\)')

async def process_single_image_with_vlm(image_url: str) -> str:
    """
    异步调用视觉大模型（VLM）处理单张图片
    这里以伪代码展示，你可以换成 qwen-vl-max, gpt-4o 等 API 的实际调用
    """
    prompt = """
    你是一个专业的数据提取专家。
    1. 如果图片是普通风景、文章封面、人物插图、二维码等无数据图，请直接回复：SKIP
    2. 如果图片是包含数据、图表、长文本的“信息图”，请深度提炼其中的核心结论和关键数据，用结构化的 Markdown 文本输出。
    绝不要输出解释性废话。
    """
    
    try:
        # 【替换为你的实际大模型 API 调用】
        # async with httpx.AsyncClient() as client:
        #     response = await client.post("YOUR_VLM_API_ENDPOINT", json={...})
        #     result = response.json()["choices"][0]["message"]["content"]
        
        # 模拟模型耗时与返回
        await asyncio.sleep(2) 
        
        # 假设大模型返回了结果
        # 实际业务中，对于封面图它会返回 "SKIP"，对于数据图它会返回 Markdown 文本
        result = "SKIP" if "qr_code" in image_url else "大模型提取的图表数据..."
        return result.strip()
        
    except Exception as e:
        # 容错处理：如果图片下载失败或模型报错，默认跳过，不影响整篇文章
        print(f"处理图片失败 {image_url}: {e}")
        return "SKIP"

async def clean_and_extract_images(text: str) -> str:
    """
    并发处理文章中的所有图片，并替换回文本中
    """
    # 1. 找出所有图片匹配项
    matches = list(IMAGE_PATTERN.finditer(text))
    if not matches:
        return text

    # 2. 并发提交给 VLM 进行处理
    tasks = [process_single_image_with_vlm(match.group(2)) for match in matches]
    # asyncio.gather 会让所有图片同时进行分析
    results = await asyncio.gather(*tasks)

    # 3. 倒序替换文本（从后往前替换，避免前面的替换导致后面的索引位置偏移）
    cleaned_text = text
    for match, vlm_result in zip(reversed(matches), reversed(results)):
        start, end = match.span()
        
        if vlm_result == "SKIP":
            # 无用图片：直接删除该图片的 Markdown 占位符
            cleaned_text = cleaned_text[:start] + "" + cleaned_text[end:]
        else:
            # 信息图：将原图片替换为提炼后的文字块
            extracted_block = f"\n> 📊 **【图表数据提炼】**：\n> {vlm_result}\n"
            cleaned_text = cleaned_text[:start] + extracted_block + cleaned_text[end:]

    return cleaned_text

@app.post("/api/v1/process_article")
async def process_article(request: ArticleRequest):
    try:
        # 执行清洗和提取管道
        final_markdown = await clean_and_extract_images(request.content)
        
        return {
            "status": "success",
            "original_length": len(request.content),
            "cleaned_length": len(final_markdown),
            "content": final_markdown
        }
    except Exception as e:
        raise HTTPException(status_code=500, detail=str(e))
```

### 💡 架构级工程优化建议（避坑指南）

由于你是在做 RAG 数据清洗管道，在实际生产环境中运行这段代码时，还需要注意以下几个工程细节：

1. **防盗链与图床限制**：
   微信公众号（`mmbiz.qpic.cn`）等平台的图片通常有防盗链机制。直接把 URL 传给大模型 API，大模型可能抓取不到图片（返回 403 Forbidden）。
   **解决方案**：在发给大模型之前，在 FastAPI 里写个小协程，用带有 `Referer` 或 `User-Agent` 伪装的 HTTP 客户端先把图片下载到内存中，然后转成 **Base64 编码**，将 Base64 直接传给大模型 API。
2. **长图切片（Image Slicing）**：
   对于极长的数据长图（比如赢商网经常发的动辄 5000 像素以上的行业报告长图），很多 VLM 会因为分辨率过高而丢失细节（发生“幻觉”或漏掉底部的几家品牌）。如果你发现 VLM 提炼不全，可以使用 Python 的 `Pillow` 库，在内存中把高度超过 2000px 的图片切成两三段，并发传给模型，最后把文本拼接起来。
3. **倒序替换策略（Reverse Replacement）**：
   请特别注意代码中的 `reversed(matches)`。这是处理字符串替换的黄金法则。如果从前往后替换，一旦第一张图片被替换成一段很长的数据文本，字符串的长度就变了，第二张图片的 `start` 和 `end` 索引就会全部错位报错。从后往前替换完美规避了这个问题。
